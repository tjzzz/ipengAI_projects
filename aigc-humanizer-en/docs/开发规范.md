# AI 辅助开发规范

> 确保多轮次、多 agent 协作时，代码质量和项目稳定性不下降。

---

## 一、Git 规范：小步提交，频繁确认

### 提交时机
每个逻辑单位完成后立即提交，不要攒多个改动一起提：

| 场景 | 做法 |
|------|------|
| 改了一个函数 | ✅ 测完就提交 |
| 改了 3 个关联文件 | ✅ 全部验证通过后一次性提交 |
| 改了不相关的 5 个功能 | ❌ 拆成 5 个提交 |

### 提交信息格式
```
<type>: <简短描述>

类型: feat / fix / refactor / docs / chore / test
示例:
  feat: 新增 Turnitin 报告检测，跳过前2页
  fix: Alipay 预创建订单返回 code=None 的问题
  refactor: 抽离 DB 连接为独立函数
```

### 提交前检查清单
- [ ] 代码无语法错误（`python3 -c "compile(...)"` 或同等方式）
- [ ] 改动涉及的功能手动测试通过
- [ ] 项目能正常启动（Flask/Django 等）
- [ ] 没有留下调试用的 print/console.log（可加 `# noqa` 标记需要的日志）

---

## 二、改前计划：先对齐，再动手

### 对于任何非 trivial 的改动，agent 应先输出计划

计划应包含：
1. **改动范围**：涉及哪些文件，改什么
2. **技术方案**：怎么改（几句话说明）
3. **风险点**：可能影响哪些功能

### 用户确认后才开始写代码

计划格式示例：
```
## 改动计划

### 范围
- app/helpers.py: extract_text_from_pdf() — 加 Turnitin 检测
- app/helpers.py: generate_file_response() — PDF 原文件输出 DOCX

### 方案
1. 读取 PDF 前2页检查 "turnitin" 关键词
2. 匹配则从第3页开始提取
3. download 时 if original_format == 'pdf' → 转为 docx

### 风险
- 非 Turnitin PDF 行为不变（第3步已测试）
- 下载参数需同步修改前端
```

## 三、改后验证：不改完就测

### 验证顺序
1. **启动检查**：项目能正常启动，无 import 报错
2. **功能验证**：走一遍改动的功能路径
3. **回归验证**：改动的功能上下游不受影响

### 验证原则
- 每改完一个文件就验证，不攒到最后
- 验证结果如实告知用户（成功/失败/异常）
- 失败时提供错误信息，不要只报"有问题"

---

## 四、配置与数据库变更：记录在案

### 配置类改动（.env / settings.py / 路径等）
- 必须告知用户具体改了什么（`ALIPAY_GATEWAY_URL → xxx`）
- 使用 CLAUDE.md 或 memory 文件记录（见第五条）

### 数据库相关
- 不要直接修改生产数据库
- 数据库路径等配置变动必须通知用户
- 涉及数据迁移的，保留备份文件（`.bak`）

---

## 五、项目记忆（Memory）：跨 session 的关键

### 需要记录的信息

| 类别 | 记录内容 | 示例 |
|------|----------|------|
| 架构决策 | 为什么某个设计这样做 | "使用 Adapter 模式隔离支付/改写引擎" |
| 配置变更 | 路径、密钥、环境变量 | "DB 路径指向项目根目录 instance/" |
| 遗留问题 | 已知但暂未修复的 bug | "沙箱网关 504，上线前需切生产" |
| 开发规范 | 项目特有的约定 | "所有 API 返回 JSON" |

### 记录格式（建议放在 CLAUDE.md 或 .claude/memory/）

```markdown
## Architecture
- 支付使用 Adapter 模式，支持 mock / alipay 切换
- 改写使用 Adapter 模式，支持 rule_based / api 切换

## Config
- DB: {project_root}/instance/aigc_humanizer.db
- Session: {project_root}/instance/flask_session/
- Uploads: {project_root}/uploads/
```

## 六、代码质量原则

### 不改动未读取的文件
- 不要修改没有 `Read` 过的文件
- 如果觉得某文件需要改，先读再改

### 优先小改动
- 加功能优先用新增函数/文件，不要重构已有逻辑
- 重构必须单独作为一个提交，不混在新功能里

### 错误处理
- 外部调用（API / 文件 IO / 网络）必须 try/except
- 错误信息要有内容，不要只返回 `"error": "失败"`
- 关键路径打日志（`logging.info/warning/error`）

### 不要引入未声明的依赖
- 新依赖必须告知用户并更新 `requirements.txt`
- 优先用 Python 标准库，减少第三方依赖

---

## 七、故障排查流程

当出现问题（如"之前能用的功能现在不能用了"）：

1. **git log** 或 **git diff** 查看最近改动
2. **git bisect** 二分定位问题提交（复杂场景）
3. **回滚实验**：`git revert` 回滚最近一次提交，确认是否修复
4. **数据库差异**：检查是否有多个数据库文件，app 是否在读取正确的那个
5. **环境差异**：检查 Python 版本、依赖版本、环境变量

---

## 八、总结：一句话原则

> **小步提交、改完就测、计划先行、文档跟上。**

把这 16 个字贴在项目根目录的 CLAUDE.md 里，任何 agent 进入时都会读到。
